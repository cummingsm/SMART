Section 3. Detailed Counts by Item

As the name indicates, this is the item level section. Details for every item are displayed here.

Another way of thinking of an item is a "Volume".

  • There will be a heading for the Volume, and one line of information per item in the Brief display, up to four lines in the Full display.
  • The "Volume" or "Sort Group" value is an eight character calculated field based on an algorithm that parses the contents of the Call Number, Item Enumeration, and Item Chron fields.
  • About the Volume / Sort Group:
    • The "Volume" is presented as a left zero-padded value so that it will sort correctly whether it is sorted as text or as a number.
    • The value '00000000' is given when there is a Call Number without an indication of a volume.
    • There are also some records that are missing a Call Number or that have some text other than an LC or Dewey Call Number.
    • Those records are assigned a sort group like "TEXTCNUM" or something like that. They will be at the end of the sorted list.
For more information about how the "Volume" is determined see the Help section "Volume Settings".

Brief Display

There are four counters from the data file displayed in the "Volume" header:

  • the number of items in this volume group that are at the home library (SMART calculated field #VOLHOME), labelled as "VLIB",
  • the number in this volume group at the WRLC SCF (SMART calculated field #VOLWRLC) labelled as "VSCF".
  • the total number of volumes either at the home library (labelled as "TLIB")
  • the total number of volumes at the WRLC SCF (labelled as "TSCF")

Volume counters are color coded as follows:

  • The VSCF count value=0 has a grey background;
  • VSCF count 1 or 2 are shaded green;
  • VSCF counts greater than 2 are shaded redorange.

The fields displayed in light green row for each item are:

  • the data field LIB, (labelled here as the "INST" column);
  • a code indicating where the volume is located (labelled as "WHERE") will be either "LIB" or "SCF"
    (this not a data field, SMART shows SCF if it finds "IS_SCF" in the
    ITEM_PERM_ID, ITEM_TEMP_ID, or MFHD_LOC_ID)
  • if WHERE says "SCF" then the background color is a darker green;
  • next is the DISP_CALL field or display call number (labelled as "CALL NO");
  • the ENUM field from item enumeration (labelled as "ENUM / VOL");
  • the item BARCODE, if any;
  • and the ITEM_STATUS_DESC field (labelled as "STATUS").

Record Selection

This system uses the BIBID or BARCODE requested as a starting point for the search. If a BARCODE was the search value it will find the WRLC BIBID linked to the BARCODE. Once a BIBID is obtained, it will search the Voyager BIB_INDEX table for more consortium records that have standard numbers in common. The system searches for matching LCCN, OCLC, ISBN, and ISSN values.

Monographs having different Editions will probably have different standard numbers. Therefore, two different editions will not likely be included in the same search result.

If there is no item or barcode linked to the WRLC Voyager record, then of course the only information the SMART report can display is the owning library name, and the holding record in the WRLC Voyager catalog. The barcode will be blank, and most often so are the call number, enumeration, and cron fields.

When the WRLC processes an item from Georgetown into the Shared Collection Facility the item barcode may be linked to the WRLC Voyager catalog record. In that case, the barcode will appear in the SMART report -- and the barcode may be used to find the record as well. For example, this Georgetown barcode 39020016393897.

When a George Mason record is in the WRLC Voyager results, the SMART program will attempt to find George Mason items at the SCF by 'screen scraping' the George Mason catalog. Items with the notation 'Charge due indefinite' will be copied and added to the SMART results. There will not be a barcode for this type of item.

Items marked with the status 'Lost', 'Missing', 'Damaged', and 'Withdrawn' are included in the SMART report. Those items are counted as if they are in the owning library collection.

Bulk Processing
The program accepts a text file containing a set of barcodes to be processed. The file should contain one barcode one each line.

Upload a list of barcodes for processing.

Step 1

  • The file should have one barcode per line.
  • Do not include more than 1,000 lines.
  • No spaces or letters should be in the file.
  • After you submit your file it will be validated.
If the validation checks pass, you will be prompted to continue processing the file in Step 2 which looks like this:

Step 2

Your file name of file has been successfully uploaded!
Enter your email address, then click the SUBMIT button below to continue processing...
 

  • After entering your email address, click the Submit button.
  • The next screen will provide an ID number for your request.
  • The system will look for requests every five minutes; depending on the number of barcodes, the number of items retrieved for each barcode, etc., your request could be ready anywhere between a minute to several minutes later.
    Estimate an average of 10 barcodes per minute. A file containing 1,000 barcodes could take (1000/10) 100 minutes, which is between an hour and half to two hours.
  • When the file has been processed you will receive an email with a link to the report.

Bulk Processing report

The bulk processing report is one text file with all of the results.

The last column on the right of the the data in the bulk processing report is the barcode that was used to retrieved the item.

If the data is imported into Excel, you could filter on the #REQUEST column to see the line or lines retrieved for that barcode.

If the program detects that it has already found the bibid that matches a barcode, it will not process it again. Instead the log will have a note saying 'already found.'

Bulk Processing errors

The program will check for the length of barcodes. If any are less than 14 characters, an error message will display, and the job will be aborted.

Use Notepad to open your barcode file; locate the problem barcodes. Either delete the line with the bad barcode, or correct it. Save the file, and return to SMART to upload the file again.

IMPORTANT notes about Bulk Processing output and working with Excel

  • Import Settings
    When importing the output from the bulk processing job into Excel, make sure you specifiy that all of the fields are of type 'Text'. The column "NORMAL_VOL must be set to type TEXT.
  • Check the titles
    After loading the data into Excel, you should sort the file by the TITLE column. Then take the time to scroll through the list of titles. You may notice that the program has already removed punctuation from the titles. That helps ensure the same title isn't sorted differently just because it does or does not end with a period character for example. If you see a title that is slightly different but obviously should be the same as the one nearby, make it match. An example would be one version of the title has the word 'The' while another does not. Remember, if a cell in one row should match the one above it, an easy fix is to click on the first cell, then drag the square from the right corner of the cell down to the next row. Excel will make the second row the same as the first.
  • About call numbers
    You may want to make a list in call number order. The column NORM_CALL is best for sorting call numbers. Unfortunately, some titles do not have call numbers. For example, periodicals / serials often do not. When those types of records are in your dataset it's not very helpful to try to sort by call number. Also, you should be aware that Gallaudet uses Dewey call numbers rather than Library of Congress call numbers. If your output doesn't have many blank call numbers, you may want to take the time to fill-in a value in NORM_CALL that you think it should be so that you could sort that way.
Consider using Microsoft Access to generate reports based on the data table.
